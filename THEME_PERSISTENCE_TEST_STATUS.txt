## How Theme Persistence Works

### Architecture: 2-Tier Storage

```
Tier 1 - LocalStorage (Immediate)
├── Stored at: window.localStorage.theme
├── Type: string ("light" | "dark")
├── Speed: ~1ms (instant)
├── Persistence: Survives page refresh
├── Limit: 5-10MB per domain
└── Fallback: Used if database unavailable

Tier 2 - Supabase Database (Persistent)
├── Table: user_preferences
├── Column: theme (text, CHECK constraint)
├── Speed: 100-500ms (async)
├── Persistence: Survives logout/login, cross-device
├── Reliability: Backed by PostgreSQL
└── Fallback: LocalStorage used if DB unavailable
```

### Theme Persistence Flow

```
Initial Load:
1. Check localStorage for theme
2. If found: use that theme
3. If not found: default to "light"
4. Load DOM with theme class
5. Fetch from database in background
6. If database has different theme: sync to database theme

Toggle Theme:
1. Click toggle button
2. setTheme() called (instant)
3. localStorage updated immediately
4. DOM class toggled
5. updateThemeInDB() called async
6. If DB fails: localStorage still works

Page Refresh:
1. localStorage read on init
2. Theme restored from localStorage
3. DOM loaded with theme class
4. Database sync in background (if needed)

Logout/Login:
1. Session cleared
2. localStorage persists (NOT cleared)
3. New user logs in
4. Database fetched for new user
5. Theme restored from database
6. UI updates
```
